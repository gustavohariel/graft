# graft in this repo

This repo has **graft** installed: a tiny bash setup that redirects `claude --worktree` so new worktrees land **outside** the repo instead of inside `.claude/worktrees/`. The reason is file-watcher cache collisions — Metro, Watchman, webpack dev servers, and IDE indexers all freak out when they see what looks like a duplicate copy of the source tree under the path they're already watching. Moving worktrees outside the main repo fixes that.

## Where worktrees live

When anyone runs `claude --worktree my-feature`, graft's `WorktreeCreate` hook fires and creates the worktree at:

```
~/.worktrees/<repo-basename>/my-feature
```

**Not** inside `.claude/worktrees/`. The exact root is set in `.claude/graft.config` as `GRAFT_WORKTREE_ROOT`. If the user asks "where is my worktree?", it's either there, or whatever they customized. `git worktree list` is always the authoritative answer.

## The four graft files

All committed so teammates inherit the setup without installing graft themselves. The runtime hook is pure bash — no dependencies, no runtime to install.

### `.claude/graft.config` — shell-sourced config

```bash
GRAFT_WORKTREE_ROOT="~/.worktrees/my-app"
GRAFT_SYMLINK_DIRS=('ios' 'android')
GRAFT_COPY_FILES=('.env' 'google-services.json')
GRAFT_SCAFFOLD=".claude/graft-scaffold.sh"
```

- **`GRAFT_WORKTREE_ROOT`** — where new worktrees are created. `~` is expanded to `$HOME` by the hook.
- **`GRAFT_SYMLINK_DIRS`** — directories that each worktree gets as a symlink back to the main repo's copy. Used for huge or platform-coupled dirs you don't want duplicated (native projects, build caches, dependency caches).
- **`GRAFT_COPY_FILES`** — gitignored files copied into each worktree. A fresh worktree is a clean git checkout, so anything that isn't tracked (env files, service-account keys, local credentials) has to be explicitly copied.
- **`GRAFT_SCAFFOLD`** — path (relative to repo root) to the user-owned bootstrap script.

This file is safe to edit directly. The hook re-reads it on every worktree creation, so changes take effect the next time the user runs `claude --worktree <name>`.

### `.claude/graft-hook.sh` — the runtime hook

~110 lines of pure `bash + git + sed`. Claude Code fires this on every `WorktreeCreate` event, passing a JSON payload on stdin (`worktree_path`, `branch`, `detach`). The hook:

1. Sources `graft.config`.
2. Runs `git worktree add` at the external location.
3. Copies the files in `GRAFT_COPY_FILES`.
4. Creates symlinks for every dir in `GRAFT_SYMLINK_DIRS` (unless overridden — see below).
5. Executes `graft-scaffold.sh` with `$1 = worktree path`, `$2 = branch`.
6. Echoes the final worktree path on stdout — that's Claude Code's contract.

**Don't edit this file directly unless the user explicitly asks.** It's meant to be regenerated by the installer (`bash <(curl -fsSL https://gustavohariel.github.io/graft/bin/install) --force`). If the user wants to change *what* happens during worktree setup, the answer is almost always "edit `graft-scaffold.sh` instead" or "edit `graft.config` instead".

### `.claude/graft-scaffold.sh` — user-owned bootstrap script

**This file is yours to edit freely.** It runs inside the newly created worktree after files are copied and dirs are symlinked. Typical contents:

```bash
#!/usr/bin/env bash
set -euo pipefail

bun install
bun openapi-ts
# Add anything a fresh worktree needs:
#   - dependency install
#   - codegen (openapi, prisma, protobuf)
#   - db seed / migrate
#   - asset linking / native module pod install
```

graft seeds this file once on first install with the detected install command (`bun install`, `pnpm install --frozen-lockfile`, `cargo fetch`, etc.) and then **never touches it again** unless the user re-runs the installer with `--force`. When the user asks "run X when a new worktree is created" or "add a setup step for new worktrees", append to this file.

Note: the scaffold runs **once at worktree creation time**, not on every `claude` session resume. If the user expects something to run on every session start, that's a Claude Code `SessionStart` hook, not graft.

### `.claude/settings.json` — registers the hook

Contains a `hooks.WorktreeCreate` entry pointing at `.claude/graft-hook.sh`. That entry is what makes graft fire at all. Other keys in the file (permissions, other hooks, env, enabled plugins) are preserved — graft merges in structurally, it doesn't overwrite.

## Per-invocation override: `GRAFT_COPY_DIRS`

Sometimes the user wants an **isolated copy** of a normally-symlinked dir for one specific worktree — for destructive updates like `gradle clean`, a native module upgrade, or experimenting with a `pod install` without clobbering sibling worktrees.

```bash
GRAFT_COPY_DIRS=android claude --worktree test-gradle-upgrade
GRAFT_COPY_DIRS=android,ios claude --worktree full-rebuild
```

For that one worktree's creation, the listed dirs are **copied** instead of symlinked. Only applies to dirs already in `GRAFT_SYMLINK_DIRS`; unknown names are silently ignored. Not persisted — the next `claude --worktree` without the env var goes back to symlinking.

Use this whenever the user needs to run something destructive against a shared native folder without affecting the rest of their worktrees.

## Quick reference: common requests

| User says | You do |
|---|---|
| "Add a step to run when a new worktree is created" | Append to `.claude/graft-scaffold.sh`. |
| "This file (e.g. `.env.staging`) isn't in my worktree" | Add it to `GRAFT_COPY_FILES` in `.claude/graft.config`. |
| "I need a fresh copy of `android/` for this worktree" | `GRAFT_COPY_DIRS=android claude --worktree <name>` — no config change needed. |
| "Where is my worktree?" | `~/.worktrees/<repo>/<name>`, or `git worktree list` for the authoritative answer. |
| "How do I regenerate the graft hook?" | `bash <(curl -fsSL https://gustavohariel.github.io/graft/bin/install) --force`. |
| "How do I uninstall graft?" | Delete `.claude/graft.config`, `.claude/graft-hook.sh`, `.claude/graft-scaffold.sh`, this skill, and the `WorktreeCreate` entry from `.claude/settings.json`. |

## Things not to do

- **Don't commit files inside `~/.worktrees/...`** as part of *this* repo. Work there, push branches from there, but don't treat it as a subdirectory of the main repo — it isn't.
- **Don't edit `graft-hook.sh` by hand** unless the user asks specifically. If you want different behavior at worktree creation, edit `graft.config` or `graft-scaffold.sh`.
- **Don't assume `graft-scaffold.sh` ran on session resume** — it only runs at worktree creation. Anything that needs to run every session is a Claude Code `SessionStart` hook.
- **Don't remove the `WorktreeCreate` entry from `.claude/settings.json`** without also removing the other graft files. Without that entry, `claude --worktree` silently falls back to default (inside-the-repo) worktree creation, which is what graft was installed to avoid.
